Pracę podzielono \emph{pionowo}: każdy mechanizm semantyczny klasy~$\mathit{DS4}$
(model dziedziny, akcje proste, akcje złożone, proces i~kwerendy, formuły,
interfejsy) ma jednego właściciela, prowadzącego go od~formalizacji w~teorii,
przez implementację modułów, po~testy; granice części pokrywają się z~granicami
modułów projektu \texttt{Ds4.Core}.

Wkład zestawiono w~czterech punktach kontrolnych: \textbf{PK1} (teoria wstępna,
26.03), \textbf{PK2} (pełna teoria, 23.04), \textbf{PK3} (prototyp aplikacji,
12.05) oraz \textbf{PK4} (wersja końcowa: teoria, aplikacja, testy,
dokumentacja, 11.06). Poza ogólną koordynacją (W.~Jasiński) wyróżniono dwie
role wiodące: \textbf{Krzysztof Gryboś} --- lider części teoretycznej (PK2)
oraz \textbf{Jarosław Bieniek} --- lider części programistycznej (PK3). Wkład
wszystkich członków był zrównoważony.

\section*{Podział prac według punktów kontrolnych}

\subsection*{Wojciech Jasiński (koordynator)}
\textbf{Obszar:} warstwa API i~integracja (\texttt{Api/Ds4Facade}, \texttt{Api/Contracts}), struktura repozytorium, przykłady demonstracyjne i~wydanie aplikacji.
\begin{itemize}\setlength\itemsep{0.1em}
  \item \textbf{PK1:} koordynacja zespołu, ustalenie zakresu klasy $\mathit{DS4}$ (warunki Z1--Z7) i~struktury dokumentu teorii wstępnej; podział mechanizmów semantycznych między członków.
  \item \textbf{PK2:} złożenie pełnej teorii w~\LaTeX{}, ujednolicenie notacji ($\Sigma$, $\Sigma_0$, $\Res$) między częściami, przygotowanie wydruku.
  \item \textbf{PK3:} szkielet rozwiązania i~układ projektów (\texttt{Ds4.Core}/\texttt{Cli}/\texttt{Gui}/\texttt{CrossGui}), fasada \texttt{Ds4Facade} spinająca parser i~semantykę w~jeden punkt wejścia, zestaw przykładów demonstracyjnych.
  \item \textbf{PK4:} reorganizacja teorii do~wersji~v2 wg~uwag recenzji, redakcja dokumentu technicznego, ujednolicenie kontraktu wyniku (TAK/NIE, uzasadnienie, $|\Sigma|$/$|\Sigma_0|$, trace, opis grafu DOT), kompilacja i~wydanie aplikacji wieloplatformowej, scalanie gałęzi i~higiena repozytorium.
\end{itemize}

\subsection*{Krzysztof Gryboś (lider części teoretycznej)}
\textbf{Obszar:} język kwerend i~procesów (\texttt{Semantics/ProcessEvaluator}, \texttt{Semantics/QueryEvaluator}, \texttt{Semantics/ExecutionTrace}, \texttt{Query/QueryModel}, \texttt{Parser/ProcessParser}, \texttt{Parser/QueryParser}).
\begin{itemize}\setlength\itemsep{0.1em}
  \item \textbf{PK1:} wstępny opis języka kwerend i~rodzajów pytań (\textit{executable}, \textit{accessible}, cel), pojęcie procesu jako sekwencji akcji.
  \item \textbf{PK2:} formalna semantyka \textit{necessary}/\textit{possibly} dla wielu stanów początkowych, definicja procesu i~śladu wykonania, kwantyfikacja po~$\Sigma_0$.
  \item \textbf{PK3:} silnik procesów i~kwerend --- budowa drzewa wykonań (poziom~$i$ to $\Res(\mathbb{A}_i,\cdot)$), oznaczanie gałęzi uciętych ($\Res=\emptyset$) oraz ewaluacja czterech wariantów pytań na~tym drzewie.
  \item \textbf{PK4:} korekta semantyki \textit{possibly} do~postaci ,,dla każdego $\sigma\in\Sigma_0$ istnieje ścieżka'' (kwantyfikacja uniwersalna po~$\Sigma_0$), rozdzielenie \textbf{after}/\textbf{observable} jako warunków ważności $M2$ zamiast filtrów~$\Sigma_0$, testy procesów i~wszystkich wariantów kwerend.
\end{itemize}

\subsection*{Jarosław Bieniek (lider części programistycznej)}
\textbf{Obszar:} parser dziedziny i~obsługa wejścia (\texttt{Parser/DomainParser}, \texttt{Parser/ParseException}) oraz interfejs wiersza poleceń (\texttt{Ds4.Cli/Program}).
\begin{itemize}\setlength\itemsep{0.1em}
  \item \textbf{PK1:} przegląd algorytmiczny --- zarys generacji modelu i~ewaluacji, identyfikacja głównych kroków przetwarzania.
  \item \textbf{PK2:} pseudokod algorytmów ($\Res$, wykrywanie konfliktów, ewaluacja kwerend) w~części teoretycznej.
  \item \textbf{PK3:} parser dziedziny (fluenty, akcje proste, zdania \textbf{causes}/\textbf{releases}/\textbf{impossible}/\textbf{initially}/\textbf{always}), wczytywanie plików wejściowych oraz interfejs wiersza poleceń (\texttt{Ds4.Cli}).
  \item \textbf{PK4:} obsługa błędnego wejścia i~czytelne komunikaty (\texttt{ParseException}), rozpoznawanie nazw akcji podobnych do~słów kluczowych, ujednolicenie składni słownej operatorów, przeniesienie algorytmów do~dokumentu technicznego, testy parsera i~regresyjne (m.in.\ warunkowe \textbf{causes}).
\end{itemize}

\subsection*{Filip Filipkowski}
\textbf{Obszar:} model dziedziny i~semantyka akcji prostych (\texttt{Model/Domain}, \texttt{Model/State}, \texttt{Model/Symbols}, \texttt{Semantics/SimpleActionEngine}, \texttt{Semantics/ModelBuilder}).
\begin{itemize}\setlength\itemsep{0.1em}
  \item \textbf{PK1:} wstępny opis języka akcji i~zdań \textbf{causes}/\textbf{releases}, intuicja efektów i~inercji.
  \item \textbf{PK2:} składnia i~semantyka akcji prostych (efekty, inercja, minimalizacja zmian, zwolnienia fluentów, niewykonalność), definicja modelu dziedziny.
  \item \textbf{PK3:} silnik akcji prostych --- operator $\Res(A,\sigma)$ z~minimalizacją zbioru zmienianych literałów, filtrowanie wartościowań do~$\Sigma$ (\textbf{always}) i~zbioru początkowego $\Sigma_0$ (\textbf{initially}, \texttt{ModelBuilder}), reguły \textbf{impossible}.
  \item \textbf{PK4:} dopracowanie inercji i~ramifikacji wg~uwag recenzji, rozdzielenie roli \textbf{initially} (źródło $\Sigma_0$) od~\textbf{always} (ograniczenia integralności), testy modelu oraz $\Sigma$/$\Sigma_0$ i~akcji prostych.
\end{itemize}

\subsection*{Sebastian Rydz}
\textbf{Obszar:} akcje złożone --- konflikty i~dekompozycja (\texttt{Semantics/CompositeActionEngine}, \texttt{Semantics/ConflictDetector}, \texttt{Semantics/DecompositionGenerator}) oraz reprezentacja zmian i~wyniku przejścia używana przez detekcję konfliktów (\texttt{Semantics/LiteralChange}, \texttt{Semantics/TransitionModel}).
\begin{itemize}\setlength\itemsep{0.1em}
  \item \textbf{PK1:} wstępny opis akcji złożonych i~problemu współbieżnego wykonywania akcji prostych.
  \item \textbf{PK2:} formalizacja konfliktów między parami akcji (przypadki sprzecznych efektów \textbf{causes}/\textbf{releases}) oraz dekompozycji jako maksymalnych bezkonfliktowych podzbiorów.
  \item \textbf{PK3:} implementacja detekcji konfliktów w~stanie~$\sigma$ i~generacji maksymalnych dekompozycji; operator $\Res(\mathbb{A},\sigma)$ akcji złożonej jako suma wyników po~dekompozycjach (źródło niedeterminizmu Z3).
  \item \textbf{PK4:} doprecyzowanie definicji konfliktu wg~teorii~v2 (dwa przypadki, usunięcie reguły nakładania się), testy konfliktów, dekompozycji i~akcji złożonych.
\end{itemize}

  \item \textbf{PK2:} podsumowanie i~redakcja pełnej teorii, spójność terminologii i~odsyłaczy.
  \item \textbf{PK3:} interfejs graficzny w~dwóch wariantach (\texttt{Ds4.Gui} --- WinForms, \texttt{Ds4.CrossGui} --- Avalonia, Linux/\,macOS), generacja przestrzeni stanów (wyliczenie wszystkich wartościowań fluentów, \texttt{StateGenerator}) oraz renderowanie grafu wykonań do~formatu Graphviz~DOT (\texttt{TraceGraphRenderer}).
  \item \textbf{PK4:} testy pełnej semantyki (scenariusze end-to-end) i~źródeł GUI (\texttt{CrossGuiSourceTests}), dopracowanie prezentacji wyników (TAK/NIE, uzasadnienie, podgląd grafu).
